Critical Performance Bottlenecks

1. Parser-Blocking JS

By default, <script> stops HTML parsing entirely until it downloads & executes.

<!-- The "Stop the World" Problem -->
<script src="app.js"></script>         <!-- ‚ùå BLOCKS PARSING -->
<script src="app.js" defer></script>   <!-- ‚úÖ FIXES IT -->

2. Render-Blocking CSS

Browser won't paint a single pixel until ALL CSS is downloaded and parsed.

<head>
  <!-- Browser waits for this before showing ANYTHING -->
  <link rel="stylesheet" href="styles.css">
</head>
‚ö†Ô∏è The Compound Effect CSS blocks rendering. CSS *also* blocks JS execution. JS blocks HTML parsing. It's a traffic jam.

Deep Dive: LCP (Loading)

What it measures:

Time to render the single largest visible element (usually hero image or main heading).

Common LCP Elements:

  • Hero images (<img>)
  • Images inside <svg>
  • Video posters (<video poster="...">)
  • Background images (via CSS url())
  • Block-level text nodes

üöÄ Quick Wins

  • ‚úÖ Preload the LCP image in <head>.
  • ‚úÖ Use AVIF or WebP formats.
  • ‚úÖ Defer non-critical JS/CSS.
  • ‚úÖ Use a CDN.

Deep Dive: INP (Responsiveness)

Replaces First Input Delay (FID). It measures the entire interaction lifecycle, not just the start.

1. Input Delay

Waiting for the main thread to free up from other tasks.

2. Processing Delay

Time spent running your actual JavaScript event handlers.

3. Presentation Delay

Time for browser to calculate layout and paint the next frame.

Reality Check: A page can have a "Good" FID score but a "Poor" INP score if your event handlers are slow!

Deep Dive: CLS (Stability)

The "jumping page" problem. Frustrating users by moving buttons under their fingers.

Common Culprits:

  • Images/Videos without width & height.
  • Ads/Iframes injecting dynamically.
  • Web fonts swapping (FOUT/FOIT).
  • Banners animating in at the top.

The #1 Fix: Reserve Space

<!-- ‚ùå BAD: Browser doesn't know the size until download starts -->
<img src="hero.jpg" alt="Hero">

<!-- ‚úÖ GOOD: Browser reserves 800x600px immediately -->
<img src="hero.jpg" alt="Hero" width="800" height="600">

Modern CSS (aspect-ratio) also helps immensely here.

Optimizing JS: async vs. defer

<script>

BLOCKS PARSING

Stops everything to download & run.

<script async>

PARALLEL DOWNLOAD

Runs as soon as it finishes. Order NOT guaranteed.

<script defer>

PARALLEL + ORDERED

Runs after HTML parses. Respects order.

Attribute Execution Time Use Case
async Whenever it finishes downloading. Independent scripts (Analytics, Ads).
defer After HTML parsing is complete. Everything else (App logic).

The fastest JavaScript is the JavaScript that is never downloaded.

Critical CSS Strategy

The Goal Render the visible "above-the-fold" content without waiting for the main stylesheet to download.
  1. Extract minimum CSS for visible viewport.
  2. Inline it directly in <head>.
  3. Load the rest of the CSS asynchronously.
<head>
  <style>
    /* Critical CSS inlined here for instant render */
    header { background: blue; height: 60px; }
    .hero-text { font-size: 3rem; font-weight: bold; }
  </style>

  <!-- Load full CSS without blocking render -->
  <link rel="stylesheet" href="full.css" media="print" onload="this.media='all'">
</head>

Optimizing CSS: Selector Performance

Key Concept: Browsers match CSS selectors from RIGHT TO LEFT.

üê¢ SLOW

nav .menu .item a { ... }

Browser finds EVERY <a> tag on the page first, then checks if it's inside .item, then .menu, etc.

üêá FAST

.nav-link { ... }

Browser finds elements with this specific class immediately. No walking up the DOM tree.

Use flat, simple, class-based selectors (like BEM or Tailwind).

CSS Architecture Showdown

Runtime CSS-in-JS

(e.g., Styled Components, Emotion)

  • JavaScript must parse and inject styles on the client.
  • High Cost: Increases JS bundle size and blocks main thread during rendering.
  • Negatively impacts INP scores.
‚ö†Ô∏è Performance Penalty

Build-Time CSS

(e.g., Tailwind CSS, vanilla-extract, CSS Modules)

  • CSS is generated into static .css files during build.
  • Zero Cost: No JS runtime overhead in the browser.
  • Browser handles it natively and fast.
üèÜ Performance Winner

Optimizing Assets: Responsive Images

Don't send a 4K desktop image to a mobile phone.

1. Resolution Switching (srcset)

You give options, browser chooses based on screen density and current width.

<img src="medium.jpg"
     srcset="small.jpg 480w,
             medium.jpg 800w,
             large.jpg 1200w"
     sizes="(max-width: 600px) 100vw, 800px"
     alt="Responsive image">

2. Art Direction (<picture>)

You force a different image crop for different screens (e.g., wide banner vs square mobile).

<picture>
   <source media="(min-width: 1024px)" srcset="wide-banner.jpg">
   <source media="(max-width: 1023px)" srcset="square-mobile.jpg">
   <img src="wide-banner.jpg" alt="...">
</picture>

The Lazy Loading Anti-Pattern

‚ö†Ô∏è

NEVER lazy-load above-the-fold content!

Lazy loading your hero image intentionally delays your LCP by seconds.

‚ùå BAD (For LCP Images)

<img src="hero.jpg" loading="lazy">

Result: Browser waits until layout is finished to even START downloading this.

‚úÖ GOOD (For LCP Images)

<img src="hero.jpg"
     loading="eager"
     fetchpriority="high">

Result: Browser downloads this immediately, often before standard CSS.

Web Font Optimization

The Modern Font Stack Strategy:

  1. 1 Format: Use WOFF2 only. It's all you need now.
  2. 2 Self-Host: Eliminate the extra DNS lookup/TLS handshake to Google Fonts.
  3. 3 Preload: Tell browser about critical fonts early.
  4. 4 Subset: Don't ship Cyrillic characters if you only need Latin.

Fixing Font-Based CLS

The Problem: Custom font takes 1s to load. Fallback font (Arial) is narrower. When custom font loads, text gets wider and pushes everything down. JUMP! (CLS Penalty)

The Solution: size-adjust CSS descriptor. We can mathematically stretch Arial to take up the exact same space as our custom font while we wait.

/* 1. The font we WANT */
@font-face {
  font-family: 'Poppins';
  src: url('poppins.woff2');
}

/* 2. The adjusted fallback */
@font-face {
  font-family: 'Poppins-Fallback';
  src: local('Arial');
  /* Magic numbers from tools like 'fallback-font-generator' */
  ascent-override: 105%;
  size-adjust: 102%;
}

/* 3. Use the stack. Zero layout shift on swap! */
body {
  font-family: 'Poppins', 'Poppins-Fallback', sans-serif;
}

Network: CDN Fundamentals

üåç

Latency is largely governed by the speed of light. We can't change physics, so we must reduce distance.

Why CDN?

  • ‚ùå
    Without CDN: User in London requests data from New York server.
    ~150ms round trip minimum.
  • ‚úÖ
    With CDN: User in London hits a cached edge node in London.
    ~10ms round trip.
The easiest + biggest global performance win.

Network: HTTP Evolution

HTTP/1.1

Like a single checkout lane at a grocery store. One item at a time.

Head-of-line Blocking

HTTP/2

Multiplexing! Many checkout lanes open at once over one connection.

Still TCP-blocked

HTTP/3 (QUIC)

Runs on UDP. Dropped packets don't stop other streams. Massive win for unstable mobile networks.

Truly Independent Streams

Network: Caching Strategies

Strategy 1: Immutable (Best)

For files with hashes in filenames (e.g., main.a8f3.js). They never change.

Cache-Control: max-age=31536000, immutable

Result: Browser never asks server again for 1 year.

Strategy 2: Revalidation

For index.html. We must check if it changed.

Cache-Control: no-cache
ETag: "v12345"

Result: Browser asks "Is v12345 still valid?". Server says "304 Yes" (tiny response).

Crucial: no-cache means "check with server before using". no-store means "never save this".

Network: Resource Hints

Resource hints tell the browser about things it will need soon, before it discovers them naturally in the HTML.

<link rel="preload" as="image" href="hero.jpg">
<link rel="prefetch" href="/checkout.js">
<link rel="prerender" href="/next-page-url">
HintPriorityUse Case
preload HIGH (Current Page) Critical assets hidden deep in CSS, or the LCP image.
prefetch LOW (Next Page) Assets for the likely next route (e.g., user is hovering a link).
prerender LOW (Next Page) Load & render entire next page in hidden tab. Extremely powerful, use with caution.

Architecture: The Rendering Spectrum

CSR (Client-Side)

Browser gets empty HTML, JS builds the whole app.

‚ùå Slow initial load (LCP). Bad for SEO.

SSR (Server-Side)

Server builds full HTML for every single request.

‚úÖ Fast LCP, Great SEO. ‚ùå Higher server load/latency.

SSG (Static)

HTML built once at deploy time. Served from CDN.

‚úÖ Fastest possible. ‚ùå Data can get stale.

ISR (Incremental)

Static pages that regenerate in the background after a timeout.

üèÜ Best of both worlds.

Advanced: Web Workers & The Main Thread

The Problem: JavaScript is single-threaded. If you run a heavy calculation, the UI completely freezes until it's done.

The Solution: Web Workers. They run on a separate thread, leaving the main thread free to handle user input and animations.

üëá Live Demo

Try to freeze the spinning square below.

I am a CSS animation on the main thread.

How it works:

// main.js
const worker = new Worker('heavy-task.js');
worker.postMessage({ data: 123 });
worker.onmessage = (e) => {
  updateUI(e.data.result); // UI stays smooth!
};

// heavy-task.js (Runs in background)
onmessage = (e) => {
  const result = doMassiveCalculation(e.data);
  postMessage({ result });
}

Advanced: Progressive Web Apps (PWAs)

A PWA uses a Service Worker to act as a network proxy. It sits between your app and the internet.

Cache-First Strategy (The "Instant Load" Secret)

self.addEventListener('fetch', (event) => {
  // Intercept EVERY network request
  event.respondWith(
    // Check cache first
    caches.match(event.request).then((cachedResponse) => {
      // Return cached file immediately if found, OR go to network
      return cachedResponse || fetch(event.request);
    })
  );
});

Result: Second visits load nearly instantly, even while offline.

The Future is fast

New Browser APIs

  • View Transitions API: Native, smooth page transitions without single-page-app complexity.
  • Speculation Rules API: Next-gen, smarter way to prefetch future pages.
  • Priority Hints: Fine-grained control with fetchpriority="high/low".

Emerging Patterns

  • Islands Architecture (Astro): Ship 0kb of JS by default, only hydrate interactive "islands" when needed.
  • Resumability (Qwik): Instant interactivity by serializing application state into HTML, eliminating standard hydration.
